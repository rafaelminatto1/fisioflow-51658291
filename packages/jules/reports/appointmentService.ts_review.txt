### ⚡ Performance Considerations

1.  **N+1 Query in `updateAppointment` for Patient Data Refresh:**
    *   **Observation:** After updating an appointment via `appointmentsApi.update`, there's a subsequent `await appointmentsApi.get(id)` call to "refetch details to complete information" (specifically patient name and phone). This is an N+1-like issue if `updateAppointment` were ever called in a batch loop, or simply an extra network roundtrip.
    *   **Suggestion:** Ideally, the `appointmentsApi.update` endpoint should return the complete updated appointment object, including denormalized patient details, to avoid this extra `GET` call. If the backend cannot provide this, the current robust fallback is acceptable, but be aware of the performance implication for individual updates. If many updates are processed in quick succession (e.g., in a bulk operation), this could lead to noticeable latency.

2.  **N+1 Query in `cancelAllAppointmentsForDate`:**
    *   **Observation:** This method first lists all appointments for a given date (`appointmentsApi.list`) and then iterates through them, calling `appointmentsApi.cancel(appt.id)` for each one. This results in `1 (list) + N (cancel)` API calls, which can be very inefficient if there are many appointments.
    *   **Suggestion:** This is a classic N+1 scenario. A significant performance improvement could be achieved if the backend provided a **batch cancellation endpoint**, e.g., `appointmentsApi.cancelBatch({ ids: ['id1', 'id2', ...] })` or even `appointmentsApi.cancelByDate(date)`. If a batch endpoint is not available, the current approach is the only option, but its performance will degrade linearly with the number of appointments.

3.  **Potential N+1 for `syncFinancialTransaction`:**
    *   **Observation:** The `syncFinancialTransaction` method calls `FinancialService.findTransactionByAppointmentId` to check for existing transactions. While currently called for single appointments (after `create` or `update`), if this method were ever adapted for batch processing, this `find` call would become an N+1.
    *   **Suggestion:** For current usage, it's fine. If batch financial syncing becomes a requirement, explore `FinancialService.findTransactionsByAppointmentIds(ids)` and batch update/create operations.
